## 職務要約

フューチャー株式会社に正社員として在籍し、フロントエンド・バックエンドからインフラ・SREまでを一人で横断するフルスタックエンジニアとして約9年半キャリアを歩んできました。

2019年9月よりコードキャンプ株式会社へ在籍出向し、Ruby on Rails / React によるアプリケーション開発を中心に、AWS を用いたインフラ・SRE 領域の改善までを一人称で担当。テックリードとして、機能開発から信頼性向上・コスト最適化・開発者体験向上まで、アプリとインフラの境界を越えてプロダクト価値の最大化に貢献しています。

新卒入社後の3年間は技術部隊のインフラチームに所属し、公共・ヘルスケア・保険・銀行など4つのプロジェクトでインフラの設計・構築を担当。この基盤の上に、出向後はアプリケーション開発まで領域を広げ、横断的に動けることを強みとしてきました。

また、直近は生成AI活用プロジェクトのPLも兼務し、社内の生成AI環境整備やAIによる業務効率化を計画・推進しています。

## スキルサマリ

フロントエンド・バックエンドからインフラ・SREまでを一人で横断できる点が強みです。アプリ開発を主軸としつつ、その土台となる基盤・運用まで自ら設計・改善できます。

### アプリケーション開発（フロント / バックエンド）

- バックエンド：Ruby on Railsを用いた機能の設計・開発が可能です。PdMと協働した仕様策定から実装・リリース・運用まで一貫して担当できます。
- フロントエンド：React（TypeScript含む）を用いた複数コンポーネントにわたる機能開発が可能です
- 生成AI活用：LLM API（OpenAIなど）を組み込んだ機能の要件定義・技術選定・実装を経験してきました。
- 品質：E2Eテスト・ユニットテストの実装、rubocopやESLintによるコード品質管理の仕組みの整備、CI/CDパイプラインの設計・運用に対応できます。
- PJ管理：小規模PJのリード経験があり、PdM・エンジニアを巻き込んだ仕様策定やタスク管理に対応できます。

### インフラ・SRE

- テックリード：インフラ・SRE領域において、チームやプロダクトの状況を鑑みた施策の立案から、設計、実装までリード可能です。
- インフラ構築・運用：パブリッククラウド（AWSなど）を用いたインフラアーキテクチャ（EC2、ECS Fargate、RDS、VPC、ALBなど）の設計・構築・運用改善を担当できます。
- IaC：構成管理ツール（CloudFormation、Ansibleなど）を用いたコード化によるインフラ構築・運用が可能です。
- CI/CD：CircleCI、GitHub Actionsを用いたCI/CDパイプラインによるテスト・ビルド・デプロイの設計・実装・運用を主担当として実施できます。
- 監視・セキュリティ：Mackerel、AWS WAF、Security Hubなどを用いた監視・セキュリティ対策の設計・実装・運用を経験してきました。Dependabotによるライブラリの脆弱性管理も運用経験があります。

##### 【2】RSpec（System Spec）を用いた E2E テストの実装

担当期間：2023年後半　役割：テックリード（テスト設計・実装・運用整備）

**背景・課題：**

- 広範に影響のあるリリース（大規模な機能開発やRailsのアップデートなど）後に障害が発生するケースが年に3〜4回程度発生しており、リリース前の品質担保の仕組みが必要な状態だった。
- 上記における対策が、PdMによるリグレッションテストとなっていたが、ケースの網羅性やテストの自動化が不十分で、自動的かつ継続的な品質チェックの仕組みとしては機能していなかった。

**取り組み：**

- E2Eテストを導入することで、継続的かつ自動的にリリース前の品質担保がされる状態を目指すことを起案し、テックリードとして設計・実装・運用整備を一貫して実施。
- PdMを巻き込んでケース別の障害レベルの設定を行い、クリティカルなものをE2Eテスト対象として選定。
- ツールとしては、RSpec と統合されており Rails アプリの E2E テストでメジャーな System Spec を採用。
  - 技術選定観点をまとめた記事：https://qiita.com/bython/items/4d79ade2e8da5378e8b6
- テストの実装およびCI パイプラインへの組み込みによるテストの自動化を行い、E2Eテストをpassした時のみmasterブランチへのマージが許可されるように設定。リリースにおいてE2Eテストによる品質が担保されている状態を実現。
- 他メンバーにはドキュメントとプルリクを元に口頭ベースでナレッジを共有。

**成果：**

- 主要な導線を E2E で網羅し、年に3〜4回起きていたリリース後の機能障害を年0〜1回程度に抑制。プロダクトの信頼性向上に貢献。
- 現在においても運用が継続されており、他メンバーによるケース追加も継続的に行われている。

`RSpec（System Spec）` `GitHub Actions` `Ruby on Rails`

##### 【3】EC2 中心の構成を ECS Fargate へ移行・運用基盤の整備

担当期間：2023年〜2024年　役割：テックリード（技術選定・設計・実装・運用定着を一貫担当）

**背景・課題：**

- EC2ベースの構成でOSパッチ適用などの保守運用が継続的に発生しており、また、インプレースデプロイのビルド負荷によるリリースタイミングの調整が発生する点や単一サーバ運用による単一障害点も課題だった。
- 会社の事業としてtoB向けに事業をピポットしていく中で、障害の影響度も高まっており、信頼性向上のためのアーキテクチャ刷新が必要な状態だった。
- また、本番リリース前の検証環境がステージング環境しか存在せず、リリースタイミングが重なる際には検証環境の利用調整が発生していた。

**取り組み：**

- まず、AmazonLinuxのEOLによるリプレイスのタイミングで、ブラックボックスとなっていたサーバーの設定を棚卸しし、Ansibleによって見える化。
- 次に、ローカル環境にDockerを導入し、メンバー参画時における開発環境構築工数（1〜2日）をほぼゼロに短縮しつつ、コンテナでアプリケーションが動作することを担保。
- 最後に、ECS Fargate 移行の設計・実装およびデプロイフローの再設計・実装を行い、本番環境へ切り替え。
- 合わせて、本番・ステージングと同等構成の ECS 環境を開発者ごとに立ち上げられる仕組みを構築し、エンジニアごとに専用の検証環境を提供できる状態に改善。

**成果：**

- サーバーレス構成になったことにより、OS パッチ運用工数を不要化。
- 冗長構成により単一障害点を解消し、サービスの信頼性を向上。
- インプレースデプロイのビルド負荷も解消され、リリースのタイミング調整も不要化。
- 開発者がECS環境を自律的に立ち上げられるようになり、プロダクト側への実装確認を行いやすい状態に改善。

`ECS Fargate` `EC2` `ALB` `CloudWatch` `CloudFormation` `GitHub Actions` `Docker`

##### 【4】CI/CD パイプラインの再設計

担当期間：2024年後半　役割：テックリード（CI/CD設計・移行推進）

**背景・課題：**

- GitHub Actions の機能成熟により、CircleCI と GitHub Actions の機能差が小さくなってきており、また市場でも GitHub Actions の採用が広がっていた。
- バージョン管理として使っている GitHub に CI ツールを統合することで、設定の一元化と運用負荷の削減を狙った。
- また、パイプラインが適正化されておらず、CIを無駄に実行したりキャッシュが有効活用されていないケースがあった。（fixtureの変更でassetをビルドしてしまうなど）

**取り組み：**

- 既存 CircleCI ジョブの棚卸しと要否を判定した上でGitHub Actions へ移行。
- パイプラインを適正化し、変更内容によって適切なステップ実行やキャッシュを利用するように修正。

**成果：**

- ツールを GitHub に一元化し、開発フローとの統合性を向上。
- キャッシュ活用により CI 実行時間を最大約5分短縮。

`GitHub Actions` `CircleCI` `Docker`

##### 【3】学習環境基盤の刷新（API の Rails 移行 + 開発環境の code-server 化）

担当期間：2025年10月〜2026年5月（うち2025年11〜12月は育児休業）　役割：PM兼テックリード（技術選定・設計・実装）

**背景・課題：**

- 学習者向け開発環境（オンラインエディタ）として AWS Cloud9 を採用していたが、AWS による新規アカウントでの利用制限が発表され、サービスの継続性に不安がある状態となった。
- 企業研修の提供において、オンラインエディタに依存している状態のため、代替手段の検討が必要となった。
- 加えて、学習環境を払い出す API が過去の担当者による Laravel 実装のまま残っており、社内標準の Rails と乖離して保守性に課題があった。
- これらの課題を解決するため、API の Rails への移行と、オンラインエディタの代替手段の検討・導入をリードすることとなった。

**取り組み：**

- PJ全体のリード
- 代替エディタとして複数候補を比較し、self-hosted 運用可能な code-server を採用することに決定。
- 約27,800ステップの Laravel から Rails への移行設計・実装。
- Cloud9 から code-server への移行に際しては、特定の企業でまずcode-server を利用してもらい、フィードバックをもらいながら段階的に切り替える方式を採用し、移行リスクを最小化。
- 上記実現のための社内関係者との調整・コミュニケーション。

**成果：**

- 社内標準技術への統一により API 保守の属人化を解消し、保守性を改善。
- Cloud9 廃止リスクを回避し、学習者向け開発環境のサービス継続性を確保することで、事業の継続性を担保。

`Laravel` `Ruby on Rails` `Docker` `code-server` `AWS SDK` `CloudFormation` `GitHub Actions`

## 自己 PR

### インフラからアプリまで一気通貫で見れるフルスタック性

React + Rails によるアプリケーション開発を主軸に、フロントエンド・バックエンドの機能実装を自ら推進してきました。加えて、そのアプリが動く基盤（AWS・CI/CD・監視）まで一人称で設計・運用できる点が最大の強みです。
「アプリだけ」「インフラだけ」ではなく両領域を地続きで見られるため、技術選定や設計において、アプリとインフラ双方の観点を踏まえた現実的な最適解を追求できます。

### アプリの土台を作れるテックリードとしての実行力

コードキャンプでは、機能開発に加えて、AWSを活用したインフラ構築・運用、信頼性向上・コスト最適化・開発者体験向上まで、プロダクト基盤の技術選定・設計・改善をテックリードとして主導してきました。
アプリケーションエンジニアでありながら、その下回りの基盤まで自分で作り切ってきた実績は、プロダクトの価値最大化にコミットできるエンジニアとしての強みになっています。

### エンジニアリングを目的化せず、事業・プロダクトの価値最大化を追求

自身が担当する実装領域においては、与えられた要件をただ実装するのではなく、プロダクトや組織にとって何が本当に必要かを考えながら、仕様の提案を含めてアウトプットしています。
また、学習環境基盤の刷新や生成AI活用プロジェクトなど、案件によっては自らディレクションやプロジェクトマネジメントも担いながら、事業・プロダクトの価値最大化にコミットしています。

## 今後の志向

フロントエンド・バックエンド・インフラを横断できる強みを軸に、レイヤを問わず会社・組織・プロダクトの価値が最大化されるよう貢献していきたいです。特定の領域に閉じず、プロダクトの課題に応じてアプリ開発にもインフラ・基盤にも入り込み、幅広く手を動かせる点を活かしたいと考えています。

そのうえで、これまで力を入れてきた信頼性・基盤領域については引き続き深めていきたいと考えています。監視・アラート整備や E2E テストによる品質担保に加え、今後は SLI / SLO の策定やエラーバジェットの運用など、信頼性を定量的に扱う取り組みにも踏み込んでいきたいです。

また、AI活用の領域についても、生成AIを活用した業務効率化やプロダクト価値向上の取り組みを継続していきたいと考えています。特に、LLM API を活用したプロダクトへの機能実装や、n8n / Dify などのワークフロー自動化ツールを活用した業務効率化の推進に注力していきたいです。
