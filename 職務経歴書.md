##### 【4】CI を CircleCI から GitHub Actions へ移行・最適化

担当期間：2024年後半　役割：テックリード（CI/CD設計・移行推進）

**背景・課題：** GitHub をバージョン管理に利用しており、CI も GitHub Actions に統一することで管理の一元化を狙えること、CIツールのデファクトスタンダードが GitHub Actions になりつつあったことから、CI を CircleCI から GitHub Actions に移行することを決定。併せて、並列化やキャッシュ戦略の見直しなど、GitHub Actions の特性を活かした最適化も実施。

**担当業務：**

- 既存CircleCIジョブの棚卸しと要否判定
- GitHub Actionsへの移行設計・実装（並列化・キャッシュ戦略の見直しを含む）

**成果：** GitHub への一元化により、開発フローとの統合性を向上。デファクトスタンダードなツールへの移行により、設定の可読性・エコシステム活用の幅が改善。ワークフローの最適化により、CIの実行時間も短縮。

`GitHub Actions` `CircleCI` `Docker` `Ruby on Rails`
:::

:::project

## 自己 PR

### インフラ・SRE 領域のテックリードとしての実行力

コードキャンプでは、インフラ・SRE 領域のテックリードとして、技術選定から設計・実装・運用まで一手に引き受けて推進してきました。この経験を活かして、貴社のインフラ基盤の信頼性の向上や開発体験の改善に貢献できると考えています。

**事例**

- AWS WAF / Security Hub の導入と運用整備
- MySQL → BigQuery の ETL 基盤構築（Embulk + digdag）
- CloudFormation + Ansible によるインフラ管理の標準化
- EC2 → ECS Fargate 移行、CircleCI → GitHub Actions 移行
- RSpec System Spec / Capybara による E2Eテスト導入

### フルスタックな技術領域

インフラ・SRE 領域のテックリードでありつつ、Rails + LLM API によるソースコード添削機能の開発など、アプリケーション開発もフロントからバックエンドまで自ら実装して推進してきました。インフラとアプリの両軸を一人称で推進できるため、技術選定や設計において、両者の観点を踏まえた最適解を追求できます。

### エンジニアリングを目的化せず、事業・プロダクトの価値最大化を追求

自身が担当する実装領域においては、与えられた要件をただ実装するのではなく、プロダクトや組織にとって何が本当に必要かを考えながら、仕様の提案を含めてアウトプットしています。
また、生成AI活用プロジェクトなど、案件によっては自らディレクションやプロジェクトマネジメントも担いながら、事業・プロダクトの価値最大化にコミットしています。

## 今後の志向

インフラ・SRE 領域を軸としつつ、レイヤを問わず引き続き会社・組織・プロダクトの価値が最大化されるよう貢献していきたいです。
